SQL INJECTION

Imagine, you are now running a bank. You let people in your bank for checking how much money is in their account, saving and withdrawing money. However, one day you let a weird guy come in, he literally goes towards your safe and takes all the money in your bank. How can this happen?

SQL, which stands for "Structured Query Language," is used for operation databases. When users can structure appropriate SQL codes to the database in the server (such as, "show/delete/modify all the usernames and passwords") and perform the SQL code successfully as a client, SQL injection happens, giving the clients unexpected authority to operate database.

SQL injection happens so common: there is a chance to trigger this vulnerability in a username / password input form in a few websites (if you are inputing a SQL code as your username and this gets interpreted in the server, HEY, DON'T DO THAT ON PURPOSE). So... how can we prevent that? The best strategies that web designers apply are filtering SQL key words and punctuations, such as a "=".

CROSS SITE SCRIPTING

Interestingly, "CSS" should be given as a name of cross site scripting as well. Since we already have "CSS" for our cascading style sheets, let's just call it "XSS."

XSS happens when attackers inject scripts (such as javascript) into the webpage and causes the leak of data (such as cookie) of others who are accessing the webpage as well. Don't get me wrong, "cookie" here does not refer to the real edible cookie, instead it is a piece of data stored in the client side. This data creates convinience such that you don't have to type your password on cetain webpage every time.

However, attackers may use some tools called XSS platforms (it's usually another programmed website) to steal your cookie, therefore the attackers can have everything YOU HAVE in this website. Giving a real example, if a XSS script goes into the log of a website and a logged-in administrator is checking out the what messes are there in the log, something awful will happen: the XSS script in the log re-directs the administrator to the XSS platform, during this unexpected visiting, the cookie of the administrator will be automatically stolen and stored by the XSS platform. Once the cookie, including the administrator's password and username, is leaked, the highest authority of the website can be accessed by the attackers.

To avoid XSS, don't easily let the users input "<>" into your webpage, and frequently checks somewhere that can store data (such as the log)! Also, since lots of people use existing CMS (contents management system) to save the time instead of making their own websites and functions, don't forget to check what're some existing vulnerabilities of the CMS that you're using!
